## Environment Variables

| Variable | Service | Required | Default | Description |

## Deploy on Vercel + Fly.io (Recommended)

This split gives you the best of both worlds: Vercel's global edge network for the Next.js frontend, and Fly.io's container platform for the Python backend with WebSocket support.

### Prerequisites

- [Vercel account](https://vercel.com)
- [Fly.io account](https://fly.io) + `flyctl` CLI installed (`curl -L https://fly.io/install.sh | sh`)
- GitHub repo with this codebase pushed

### Step 1 — Deploy Backend on Fly.io

```bash
cd backend

# Launch the app (creates fly.toml)
fly launch --name confluence-backend --no-deploy

# Set environment variables
fly secrets set \
  CORS_ORIGINS=https://confluence-frontend.vercel.app \
  LOG_LEVEL=INFO

# Optional: provision Redis on Fly.io
fly redis create --name confluence-redis
# Copy the returned REDIS_URL and set it:
fly secrets set REDIS_URL=redis://default:<password>@confluence-redis.flycast:6379

# Deploy
fly deploy
```

Verify the backend is running:

```bash
curl https://confluence-backend.fly.dev/health
# Expected: {"status":"ok","version":"0.1.0"}
```

#### `fly.toml` (backend)

This is auto-generated by `fly launch`. Key settings to verify:

```toml
app = "confluence-backend"
primary_region = "iad"

[build]
  dockerfile = "Dockerfile"

[http_service]
  internal_port = 8000
  force_https = true
  auto_stop_machines = true
  auto_start_machines = true
  min_machines_running = 0

  [http_service.concurrency]
    type = "connections"
    hard_limit = 50
    soft_limit = 25

[[vm]]
  memory = "1gb"
  cpus = 2
  cpu_kind = "shared"

[checks]
  [checks.health]
    port = 8000
    type = "http"
    interval = "30s"
    timeout = "5s"
    path = "/health"
```

### Step 2 — Deploy Frontend on Vercel

```bash
# From the repo root, or via Vercel dashboard
cd frontend
vercel --prod
```

Or via the Vercel dashboard:

1. Go to [vercel.com](https://vercel.com) → **New Project**
2. Import your GitHub repo
3. Configure:
   - **Framework Preset**: Next.js
   - **Root Directory**: `frontend`
   - **Build Command**: `npm run build` (default)
   - **Output Directory**: `.next` (default)
4. Add Environment Variable:
   - `NEXT_PUBLIC_API_URL` = `https://confluence-backend.fly.dev`
5. Click **Deploy**

After deployment, note your Vercel URL (e.g., `https://confluence-frontend.vercel.app`) and update the backend CORS:

```bash
cd backend
fly secrets set CORS_ORIGINS=https://confluence-frontend.vercel.app
```

### Step 3 — Verify End-to-End

1. Open `https://confluence-frontend.vercel.app/app`
2. Select an algorithm and dataset, click **Run**
3. Verify the heatmap renders (backend is responding)
4. Open browser DevTools → Network tab, confirm API calls go to `confluence-backend.fly.dev`

### Fly.io Cost Notes

- **Free tier**: 3 shared-cpu-1x VMs with 256MB RAM (enough for the backend)
- **Redis**: Fly Redis is free for a single instance
- **Paid**: ~$5-10/month for a 1GB VM if you need more memory for large datasets

### Vercel Cost Notes

- **Hobby plan (free)**: Unlimited static sites, 100GB bandwidth
- **Pro plan ($20/month)**: If you need more bandwidth or team features

## Deploy on Vercel + Railway

Railway is an alternative to Fly.io if you prefer a simpler dashboard experience.

### Backend on Railway

1. Go to [railway.app](https://railway.app) → **New Project**
2. **New → Database → Redis** — note the internal URL
3. **New → GitHub Repo** → select your repo
4. Configure the backend service:
   - **Root Directory**: `backend`
   - **Start Command**: `uvicorn app.main:app --host 0.0.0.0 --port $PORT --workers 4`
   - **Environment Variables**:
     - `REDIS_URL` = `redis://<redis-internal-url>`
     - `CORS_ORIGINS` = `https://confluence-frontend.vercel.app`
     - `LOG_LEVEL` = `INFO`
5. Note the generated URL (e.g., `confluence-backend.up.railway.app`)

### Frontend on Vercel

Same as Step 2 above, but set `NEXT_PUBLIC_API_URL` to your Railway backend URL.

## Deploy on Render (Full Stack)

### Steps

1. Push code to GitHub
2. Go to [render.com](https://render.com)

#### Redis
- **New → Redis** → note internal URL

#### Backend (Web Service)
- **New → Web Service** → Connect repo
- **Root Directory**: `backend`
- **Runtime**: Docker
- **Environment Variables**:
  - `REDIS_URL` = `redis://<redis-internal-url>`
  - `CORS_ORIGINS` = `https://your-frontend.onrender.com`

#### Frontend (Static Site)
- **New → Static Site** → Connect repo
- **Root Directory**: `frontend`
- **Build Command**: `npm ci && npm run build`
- **Publish Directory**: `.next`
- **Environment Variables**:
  - `NEXT_PUBLIC_API_URL` = `https://your-backend.onrender.com`

## Docker Compose (Production)

For self-hosted deployments (VPS, dedicated server, Kubernetes node).

```bash
# Set production env vars
export CORS_ORIGINS=https://yourdomain.com
export REDIS_URL=redis://redis:6379
export NEXT_PUBLIC_API_URL=https://api.yourdomain.com

# Build and start
docker compose -f docker-compose.yml up -d --build

# Check status
docker compose ps
docker compose logs -f backend
```

### Reverse Proxy (Nginx)

```nginx
server {
    listen 443 ssl;
    server_name yourdomain.com;

    # SSL (Let's Encrypt)
    ssl_certificate /etc/letsencrypt/live/yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/yourdomain.com/privkey.pem;

    # Frontend
    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Backend API
    location /api/ {
        proxy_pass http://localhost:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # WebSocket (training animation)
    location /ws/ {
        proxy_pass http://localhost:8000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 86400;
    }
}
```

### Caddy (Alternative)

```
yourdomain.com {
    reverse_proxy /api/* localhost:8000
    reverse_proxy /ws/* localhost:8000 {
        transport http {
            read_timeout 86400
        }
    }
    reverse_proxy localhost:3000
}
```

## Troubleshooting

| Issue | Solution |
|-------|----------|
| Redis connection refused | Ensure Redis is running; check `REDIS_URL`. App works without Redis (caching disabled). |
| CORS errors | Set `CORS_ORIGINS` to your frontend domain (comma-separated for multiple) |
| WebSocket fails behind proxy | Ensure proxy forwards `Upgrade` and `Connection` headers. See Nginx config above. |
| Build fails (frontend) | Run `npm ci` (not `npm install`); ensure `output: "standalone"` in `next.config.ts` |
| 500 on unknown algorithm | Returns 422 with validation error (expected behavior) |
| Fly.io app sleeps | Free tier auto-stops idle machines. First request takes ~2s to wake. Set `min_machines_running = 1` to prevent. |
| Vercel build fails | Ensure Root Directory is set to `frontend`, not the repo root |
| LDA fails on 2-class data | LDA reduces to 1 component for 2 classes (mathematical constraint). Result is padded to 2D. |
